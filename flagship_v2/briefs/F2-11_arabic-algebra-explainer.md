## Brief
Context: A client is commissioning a concise, two-minute educational video that teaches linear (one-variable) equations in clear, age-appropriate Arabic delivered with a genuine GCC (Gulf) accent. The piece is explicitly a MIXED format: short live-action segments featuring a boy aged 12–15, blended seamlessly with animated graphics that illustrate the algebra steps. Use the supplied reference lecture to derive the precise solving sequence (problem → isolate the variable → divide by the coefficient → state x → substitute and check). Produce four animated scenes (one per solve beat) carrying Arabic equation/step copy, interweave them IN A FIXED PEDAGOGICAL ORDER with the live-action boy footage into a single cut, lay the polished Arabic narration over the picture, and master to spec.\n\nHow the connector workflow realizes it (execution-mode honest): media_summarize [A] structures the reference lecture; the four scenes are TEXT-filled on a single user-picked Express template (search_design [C] returns the gallery, the user picks, fill_text [W] fills text only, animate_design [W] adds motion); media_enhance_speech [A] cleans the VOICE-only VO without altering the accent; video_resize [A] frame-locks every scene clip to 1920×1080 at the SAME length; and the ORDER-CRITICAL weave of the four scenes with the boy B-roll, the VO mux, the final exact-duration trim, AND the timecode RTL Arabic SRT are all done LOCALLY [L] with ffmpeg/alignment, because no connector tool sequences deterministically or emits time-aligned subtitles. The picture is pre-built to ~115–120s so only a light trim lands the exact 2:00.\n\nConstraints / deliverables: (1) Final 1080p MP4 (1920×1080, H.264), runtime exactly 2:00 ±5 sec. (2) A SEPARATE Arabic SRT subtitle file (RTL, timed to the VO, built locally). (3) Editable source / scene artifacts for future edits (the four filled Express designs and the four frame-locked per-scene animation clips, bundled via asset_copy_assets). Acceptance criteria, all enforced: spoken Arabic matches a modern GCC accent throughout (and is NOT flattened by cleanup); live-action and animation are interwoven in a fixed order with NO abrupt cuts or style inconsistencies (style breaks are a reject); and by the end a learner can solve a one-variable linear equation AND check the answer.

**Iterative chaining:** The chain is genuinely output→input at every hop. Step 1 (media_summarize [A]) converts the reference lecture into a structured solve sequence + rough transcript timings; that single artifact is the source of truth flowing into the template search (4), all four scene fills (7,8,9,10 — each pulls a different solve beat), and the LOCAL SRT build (18). Steps 2→3 turn the four supplied assets into CC URNs that later steps reference: bg_urn → inline preview (6) and the four fill_text scenes (7–10); vo_raw_urn → media_enhance_speech (15); boy_urn → the editable bundle (17) and the LOCAL weave (18). The Express sub-chain is strictly serial: search_design (4) yields the user-picked template_id reused by every fill_text; font_recommend (5) yields a font reused by every fill_text; each fill_text output (7/8/9/10) is the sole input to its matching animate_design (11/12/13/14). The four animation clips are frame-locked together by video_resize (16, same-length reframe), then asset_copy_assets (17) bundles the editable source. The cleaned VO (15) plus the step-1 transcript drive the SRT, and the frame-locked clips (16) + boy_urn (3) + cleaned VO (15) all converge in the single LOCAL ffmpeg step (18) → the ordered 2:00 master + SRT. Nothing is invented mid-chain; every consumed item is the literal output of an earlier numbered step or a named input. No connector tool is asked to do something outside its real capability (no Quick-Cut sequencing, no layout-renderer SRT).


**Why it's hard:** Long horizon: 18 ordered calls (12 connector calls across the Express-template, video/audio, and asset-storage groups) with the whole build pivoting on a single derived artifact (the step-1 lecture summary) that must stay coherent across the template search, four separate scene fills, and the local SRT. State tracking is heavy: one user-picked template_id and one font choice are reused across four fill_text calls; four filled designs map 1:1 onto four animate_design calls; four URNs from a two-step upload must be threaded to the right downstream consumers (boy_urn→weave, vo_raw_urn→enhance_speech, bg_urn→fills) — mis-binding any silently breaks the chain. Mixed execution modes raise the bar: the agent must recognize the [W] Express steps need a human gallery pick, the [A] media steps return async widgets, and that the order-critical weave + the timecode SRT are NOT connector-doable and must be done locally — a key correctness trap the original spec failed (it misused Quick Cut as a sequencer and a page-renderer as an SRT emitter). Exact-spec pressure is high and adversarially graded: runtime must land in 2:00 ±5s (picture pre-built to ~115–120s so only light trim is needed), frame must be exactly 1920×1080 (locked per-clip via same-length video_resize), the GCC accent must survive enhancement (an easy place to over-process — the VO is now a near-clean take), live-action and animation must interweave in a FIXED pedagogical order with no style break, and the on-screen math must be correct AND self-checking. The agent must make genuine content judgments (decompose the solve into four beats, write correct RTL Arabic equation copy, align the SRT) rather than just call tools — a wrong number on screen, a flattened accent, or an engagement-reordered weave fails acceptance even if every tool call 'succeeds'.
